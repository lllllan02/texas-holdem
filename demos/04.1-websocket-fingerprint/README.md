# Demo 04.1: WebSocket 身份指纹与结构化消息

这个 Demo 是 `04-websocket-basics` 的进阶版。它模拟了真实的德州扑克游戏大厅，引入了**用户身份识别**、**结构化 JSON 通信**、**私有消息单播**以及**顶号逻辑**。

## 核心目标

1. **前端指纹识别 (Fingerprint)**：
   - 使用 `localStorage` 为每个浏览器生成并持久化一个唯一的 `UserID` 和随机昵称（如 "游客_A1B2"）。
   - 验证：用 Chrome 和 Safari 分别打开页面，会被识别为两个不同的玩家；刷新页面，身份不会丢失。
2. **结构化消息协议 (JSON)**：
   - 抛弃了纯文本通信，改用 JSON。
   - 前端发送：`{"userId": "...", "username": "...", "action": "bet", "content": "100"}`
   - 后端使用常量（如 `ActionJoin`, `ActionBet`）处理路由，解析后加上服务器时间戳，再广播或单播。
3. **私有消息单播 (Unicast)**：
   - 德州扑克中，玩家的底牌是绝对私密的。
   - 提供了【👁️ 看底牌 (私有消息)】按钮，点击后服务器只会将底牌信息发送给该玩家本人的 WebSocket 连接，其他人无法收到。
4. **顶号与断线逻辑 (Kick out & Leave)**：
   - **加入/离开广播**：玩家打开页面会自动广播“加入了大厅”，关闭页面会自动广播“离开了大厅”。
   - **顶号逻辑**：如果同一个 `UserID` 在其他地方（如新标签页）登录，服务器会主动向旧连接发送一条红色的 `kick` 警告消息，并强制断开旧连接，保证账号唯一性。被踢下线的前端会**停止自动重连**，防止无限互相顶号。

## 核心难点与踩坑记录 (重点)

在实现这个 Demo 的过程中，我们解决了几类非常经典的 WebSocket 架构问题，这些也是构建工业级游戏服务器的必经之路：

### 1. 顶号逻辑的“误杀”问题
- **问题**：最初设计时，我们将身份绑定放在了“收到第一条消息”时。这导致新连接在刚建立（UserID 还是空字符串）的那一瞬间，会把上一个同样还没发消息的连接踢掉，导致不同浏览器的用户互相“顶号”。
- **解法**：**身份识别必须前置**。在 HTTP 升级为 WebSocket 之前，直接通过 URL 参数 (`ws://...?userId=xxx`) 获取身份。如果没传直接拒绝连接，确保进入 `Hub` 的每一个连接都有明确的身份。

### 2. 顶号导致的无限重连循环
- **问题**：前端有断线自动重连机制。当页面 A 被页面 B 顶号踢下线后，A 会触发 `onclose` 并自动在 3 秒后重连，这又会把 B 踢下线，导致两个页面陷入无限的互相顶号循环。
- **解法**：在前端引入 `isKicked` 状态锁。当收到后端的 `kick` 消息时，将 `isKicked` 设为 `true` 并主动关闭连接。在 `onclose` 事件中判断，如果是被踢下线的，则**终止重连定时器**。

### 3. 并发关闭 Channel 导致的 Panic
- **问题**：在顶号时，服务器会主动 `close(oldClient.Send)`。但同时，旧连接断开也会触发 `readPump` 的 `defer` 清理逻辑，导致 `Hub` 再次执行 `close(oldClient.Send)`。在 Go 中，重复关闭 channel 会引发 `panic: close of closed channel`。
- **解法**：在 `Client` 结构体中增加一个内部状态标志 `closed bool`。在 `Hub` 的单 Goroutine 主循环中，每次调用 `close` 前先检查并更新该标志，确保每个 channel 只被关闭一次。

## 目录结构
```text
demos/04.1-websocket-fingerprint/
├── server/
│   ├── main.go      # HTTP 服务入口 (监听 8081 端口)
│   ├── client.go    # 封装 WebSocket 连接，包含 JSON 解析、路由分发与单播逻辑
│   └── hub.go       # 广播中心，包含 UserID 映射和顶号逻辑
├── web-demo/
│   └── index.html   # 包含指纹逻辑和交互按钮的测试页
└── README.md
```

## 如何测试

1. **启动后端服务**：
   ```bash
   cd demos/04.1-websocket-fingerprint/server
   go run .
   ```
   *(注意：为了不和 Demo 04 冲突，此服务监听在 `8081` 端口)*

2. **打开前端页面**：
   在浏览器中打开 `demos/04.1-websocket-fingerprint/web-demo/index.html`。
   - 你会在右上角看到为你自动生成的游客昵称和 ID。
   - 尝试点击【坐下】、【下注】等按钮，或者发送聊天消息。

3. **多开与私有消息测试**：
   - 复制 URL，在**无痕模式 (Incognito)** 或**另一个浏览器**中打开（获得全新身份）。
   - 两个窗口互相点击按钮，可以看到带有颜色的动作广播。
   - 点击紫色的【👁️ 看底牌】按钮，验证只有点击者本人能收到该消息。
   - 关闭其中一个窗口，另一个窗口会收到“离开了大厅”的广播。

4. **顶号逻辑测试**：
   - 在普通模式下打开页面 A。
   - 复制 URL，在**同一个浏览器新开一个标签页 B**（共享 localStorage，身份相同）。
   - 页面 B 连接成功时，页面 A 会收到一条红色的【警告】消息并被强制断开连接！
